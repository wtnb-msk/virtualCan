# CI 手順書（GitHub Actions）

## 概要

`ubuntu-latest` ランナー上で vcan0 を使った PoC を自動実行する。
push / pull_request のたびに全ターゲットがビルド・テストされる。

---

## CI を実行するには

**特別な操作は不要。** `main` ブランチへの push または pull_request の作成で自動的に起動する。

```bash
git push origin main   # これだけで CI が走る
```

結果は GitHub の **Actions タブ** で確認できる。

---

## ワークフローの流れ（.github/workflows/ci.yml）

```
1. システム依存パッケージをインストール（can-utils、linux-modules-extra）
2. Python venv を構築（requirements.txt）
3. C++ ターゲットをビルド（g++）
4. vcan0 をセットアップ
5. ecu_cpp を実行 → scenario.yml の assertions でログをアサート
6. logs/ を Artifact としてアップロード
```

---

## 成否判定の仕組み（check_result.sh）

```bash
bash scripts/check_result.sh logs/ecu.log targets/<name>/scenario.yml
```

`scenario.yml` の `assertions` セクションを読み込み、各 `pattern` が `logs/ecu.log` に含まれるかを確認する。
すべての pattern が見つかれば PASS、1 つでも欠ければ FAIL（exit 1）で CI が失敗する。

### assertions の書き方

```yaml
assertions:
  - pattern: "[NORMAL]"
    description: "正常フレームが NORMAL と判定されること"
  - pattern: "[ABNORMAL]"
    description: "異常フレームが ABNORMAL と判定されること"
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| `pattern` | **必須** | ECU アプリが `logs/ecu.log` に出力する文字列。この文字列がログに含まれるかどうかで合否を判定する |
| `description` | 任意 | CI の結果画面に表示されるテスト名。動作には影響しない |

**実車 App に差し替えるときは `pattern` だけ変えればよい。**
`description` はわかりやすい名前に直したければ変える、という位置づけ。

```yaml
# 実車 App の出力に合わせた例
assertions:
  - pattern: "ECU_STATE: FAULT"
    description: "異常時に FAULT ステートに遷移すること"
  - pattern: "DTC: P0100"
    description: "DTC コード P0100 が記録されること"
```

### CI 結果画面での見え方

```
PASS: 正常フレームが NORMAL と判定されること
PASS: 異常フレームが ABNORMAL と判定されること
```

失敗した場合：

```
PASS: 正常フレームが NORMAL と判定されること
FAIL: 異常フレームが ABNORMAL と判定されること  (pattern not found: '[ABNORMAL]')
```

## 新ターゲットの追加方法

### 1. ターゲットディレクトリを作成

```bash
mkdir targets/my_ecu
```

### 2. main.cpp と scenario.yml を配置

### 3. scenario.yml に assertions を記載

```yaml
name: my_ecu_test

assertions:
  - pattern: "期待するログ文字列"
    description: "テスト内容の説明"

steps:
  - label: "フレーム送信"
    template: "123#0102030405060708"
    count: 5
    interval: 0.1
```

### 4. ci.yml に3ステップを追加

```yaml
- name: Build my_ecu
  run: g++ -O2 -o targets/my_ecu/app targets/my_ecu/main.cpp

- name: Run PoC (my_ecu)
  run: sudo bash run_poc.sh targets/my_ecu

- name: Check result (my_ecu)
  run: bash scripts/check_result.sh logs/ecu.log targets/my_ecu/scenario.yml
```

---

## ランナーの vcan 対応

WSL2 と異なり、ubuntu-latest ランナーはカスタムカーネル不要。
`linux-modules-extra-$(uname -r)` をインストールするだけで `modprobe vcan` が成功する。

---

## ログの確認

CI 完了後、**Artifacts** セクションから `poc-logs` をダウンロードすると `logs/ecu.log` と `logs/candump.log` が取得できる。
CI が失敗した場合はこのログを見て原因を特定する。
